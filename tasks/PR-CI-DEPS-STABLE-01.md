PR-CI-DEPS-STABLE-01
======================

Prompt (copy as-is into Codex)
------------------------------

```text
You are Codex Agent working on the LAN-Transcriber repository.

Goal
Fix the persistent runtime failure "ModuleNotFoundError: No module named 'matplotlib'" and make CI reliably catch this class of issue.
At the same time, implement stable CI lockfiles using pip-tools (Variant A) and remove the unused CPU runtime path (CUDA-only).

Context
- Production runtime can crash when importing whisperx.asr / whisperx.load_model because the import chain reaches pyannote.audio tasks that import matplotlib.
- Current docker smoke and Dockerfile import checks only do "import whisperx" which does not always trigger the failing import chain.
- The repo contains both CPU and CUDA requirements lockfiles (requirements.txt and requirements-cu121.txt). CPU is not planned to be supported.
- CI currently installs ci-requirements.txt, but the file is not generated by pip-compile (it is a hand-maintained loose constraints file). We want Variant A: ci-requirements.in is the source of truth, ci-requirements.txt is generated by pip-compile for reproducible CI.

Non-goals
- Do not add new application features.
- Do not change the pipeline logic beyond dependency and smoke-check behavior.
- Do not introduce new secrets into CI.

Implementation requirements

1) Make runtime CUDA-only (remove CPU lockfiles and usage)
- Stop using requirements.in and requirements.txt for any build or workflow.
- Update Dockerfile so runtime-lite installs requirements-cu121.txt (same as runtime-full) and remove the CPU requirements install step.
- Update docker-compose.yml default target so TRANSCRIBER_DOCKER_TARGET defaults to runtime-full (or keep runtime-lite but ensure it is CUDA-based).
- Update any workflows referencing requirements.txt to use requirements-cu121.txt or remove that step if it is unnecessary (preferred: remove).
- Remove requirements.in and requirements.txt from the repository if they are no longer referenced anywhere.
- Update README/runbook lines that refer to requirements.txt to refer to requirements-cu121.txt.

2) Add matplotlib to the CUDA requirements and re-lock
- Add "matplotlib" to requirements-cu121.in.
- Re-run pip-compile to regenerate requirements-cu121.txt:
  - pip-compile --extra-index-url=https://download.pytorch.org/whl/cu121 --output-file=requirements-cu121.txt requirements-cu121.in
- Run python -m pip check in the container build to ensure consistency (already present).

3) Strengthen smoke checks to actually catch matplotlib/whisperx import-chain issues
- Update tests/test_docker_smoke.py dependency import check to include:
  - import whisperx.asr
  - import matplotlib
- Keep it fast: do not download models, do not call inference.
- Update Dockerfile import sanity check to also import whisperx.asr (and optionally matplotlib) so failures are detected at build time as well.

4) Update diarization model id default in compose (pyannote v3)
- In docker-compose.yml, update LAN_DIARIZATION_MODEL_ID default from "pyannote/speaker-diarization@3.1" to "pyannote/speaker-diarization-3.1".
- Ensure documentation and warmup examples (if any) use the repo-name format for pyannote v3.

5) Variant A for CI requirements (pip-compile generated)
- Make ci-requirements.in the only hand-edited file for CI deps.
- Regenerate ci-requirements.txt using pip-compile so it is pinned and reproducible:
  - pip-compile --output-file=ci-requirements.txt ci-requirements.in
- Remove the hand-maintained loose-constraints content from ci-requirements.txt (replace with pip-compile output).
- Ensure scripts/ci.sh keeps installing -r ci-requirements.txt (no behavior change, just pinned deps).
- Add a lightweight CI guard step in ci.yml that fails if ci-requirements.txt is out of date relative to ci-requirements.in:
  - run pip-compile to a temp file and diff against repo ci-requirements.txt
  - do not write to repo in CI, only compare
- Keep this guard fast and deterministic.

6) Clean up duplicate/legacy workflows references
- If any legacy workflow files reference requirements.txt or duplicate the main CI, update or remove them so they do not break in CUDA-only mode.
- Ensure the primary workflow (.github/workflows/ci.yml) remains the source of truth.

7) Tests
- Update existing tests only as needed (docker smoke, CI guard).
- Run full suite: scripts/ci.sh
- Ensure docker-smoke-pr job still passes with the updated runtime-lite target and improved import checks.

Deliverables
- CUDA-only runtime: Dockerfile and compose updated, CPU requirements removed/unused
- requirements-cu121.in includes matplotlib and requirements-cu121.txt regenerated
- docker smoke import checks catch whisperx.asr and matplotlib failures
- docker-compose.yml default diarization model id fixed for pyannote v3
- ci-requirements.txt regenerated via pip-compile from ci-requirements.in, plus CI guard for drift

Success criteria
- Building the runtime image and running the docker smoke test fails if matplotlib is missing.
- In the runtime container, "python -c 'import whisperx.asr; import matplotlib'" succeeds.
- CI uses pinned ci-requirements.txt generated from ci-requirements.in and fails if the lockfile is out of date.
- CPU requirements.txt is no longer used anywhere in Dockerfile, compose, or workflows.
```
